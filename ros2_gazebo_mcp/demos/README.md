# Gazebo MCP - Demonstration Scenarios

This directory contains complete, polished demonstration scenarios for the ROS2 Gazebo MCP Server. Each demo is designed for specific audiences and use cases, with full scripts, setup guides, and recovery plans.

## Quick Start

```bash
# Navigate to a demo
cd demos/01_hello_world

# Run setup (first time only)
./setup.sh

# Review the script
cat script.md

# Practice the demo
./quick_test.sh
```

## Demo Structure

Each demo directory contains:

```
demos/XX_demo_name/
├── script.md              # Complete demo script with timing
├── setup.sh               # Automated setup and verification
├── config.yaml            # Configuration and customization
├── troubleshooting.md     # Detailed troubleshooting guide
├── quick_test.sh          # Quick pre-demo test (generated by setup)
├── start_mcp_server.sh    # MCP server launcher (generated by setup)
├── emergency_recovery.sh  # Emergency recovery script (generated by setup)
├── PRE_DEMO_CHECKLIST.md  # Pre-demo checklist (generated by setup)
├── logs/                  # Demo logs (created during demo)
└── slides/                # Optional presentation slides
```

## Usage Guide

### First-Time Setup

1. **Choose a Demo**:
   ```bash
   cd demos/01_hello_world
   ```

2. **Run Setup**:
   ```bash
   chmod +x setup.sh
   ./setup.sh
   ```

3. **Review Materials**:
   ```bash
   cat script.md              # Read the full script
   cat PRE_DEMO_CHECKLIST.md  # Review checklist
   cat config.yaml            # Customize if needed
   ```

4. **Practice**:
   ```bash
   ./quick_test.sh            # Quick system test
   ```

### Before Each Demo

Follow the generated `PRE_DEMO_CHECKLIST.md`:
- Close unnecessary applications
- Test all systems
- Have backup plans ready
- Position screens for visibility

### During Demo

- Follow `script.md` timing guide
- Stay calm if issues arise
- Use recovery plans if needed
- Engage with audience

### Emergency Recovery

If demo fails:
```bash
./emergency_recovery.sh  # Kills all processes and restarts
```

Then restart MCP server:
```bash
./start_mcp_server.sh
```

## Customization

Edit `config.yaml` in each demo to customize:
- Robot models and variants
- Movement parameters
- Obstacle configurations
- Timing adjustments
- Camera positioning
- Logging options

Example:
```yaml
robot:
  model: "turtlebot3"
  variant: "waffle"  # Change from burger to waffle

movements:
  part3_forward_distance: 3.0  # Increase from 2.0 to 3.0

obstacles:
  part4_count: 6  # Increase from 4 to 6
```

## Best Practices

### Before Presenting

1. **Practice 3+ times**:
   - Run through complete demo
   - Time yourself
   - Note any issues

2. **Record backup video**:
   ```bash
   # Enable recording in config.yaml
   logging:
     record_video: true
   ```

3. **Test on presentation hardware**:
   - Same laptop/computer
   - Same screen resolution
   - Same network conditions

4. **Prepare for questions**:
   - Review FAQ in script.md
   - Have technical details ready
   - Know recovery procedures

### During Presentation

1. **Start with context**:
   - Who you are
   - What they'll see
   - Why it matters

2. **Show, don't tell**:
   - Live demos > slides
   - Visual > verbal
   - Interactive > passive

3. **Pace yourself**:
   - Speak slowly
   - Pause after commands
   - Allow time for effect

4. **Engage audience**:
   - Make eye contact
   - Ask questions
   - Welcome interaction

5. **Handle failures gracefully**:
   - Stay calm and confident
   - Use recovery scripts
   - Have backup video ready
   - Make it a learning moment

### After Presenting

1. **Answer questions**:
   - Take time for Q&A
   - Offer to demo more offline
   - Share contact info

2. **Share materials**:
   - GitHub repository
   - Documentation links
   - Tutorial series
   - Demo videos

3. **Gather feedback**:
   - What worked well?
   - What confused people?
   - What would they change?
   - Would they recommend?

4. **Document lessons**:
   - Technical issues encountered
   - Audience reactions
   - Timing adjustments needed
   - Questions for FAQ

## Troubleshooting Common Issues

### Gazebo Won't Start
```bash
# Kill existing processes
killall -9 gz gzserver gzclient

# Clear cache
rm -rf ~/.gz/sim/*

# Restart
gz sim empty.sdf
```

### MCP Server Connection Issues
```bash
# Verify ROS2 sourced
source /opt/ros/humble/setup.bash

# Check ROS2 environment
env | grep ROS

# Restart MCP server
./start_mcp_server.sh
```

### Robot Doesn't Spawn
```bash
# Verify model files exist
ros2 pkg list | grep turtlebot3

# If missing:
sudo apt install ros-humble-turtlebot3 ros-humble-turtlebot3-description

# Check Gazebo resource paths
gz model --list
```

### Robot Doesn't Move
```bash
# Check ROS2 topics
ros2 topic list

# Verify cmd_vel topic exists
ros2 topic echo /cmd_vel

# Check controller
ros2 control list_controllers
```

## Advanced Usage

### Chaining Demos

Create a presentation with multiple demos:
```bash
# Script to run demos in sequence
./01_hello_world/quick_demo.sh    # 3 min
./02_obstacle_course/standard.sh  # 10 min
./03_multi_robot/quick.sh         # 12 min
# Total: 25 minutes
```

### Custom Variations

Create your own demo variation:
```bash
cd demos/01_hello_world
cp config.yaml config_custom.yaml
# Edit config_custom.yaml
# Load with: CONFIG=config_custom.yaml ./quick_test.sh
```

### Recording Demos

Enable video recording:
```yaml
# In config.yaml
logging:
  record_video: true
  video_path: "./recordings/"
  video_format: "mp4"
  video_quality: "high"
```
